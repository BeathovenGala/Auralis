I. Technical Requirements: Backend and Data Engineering
This section specifies the foundational architectural changes required to implement the new forecasting system, addressing the critical distinction between the academic paper's historical analysis and the requirement for a real-time, predictive tool.

I.1 Data Ingestion and Processing Pipeline
The new system requires a high-resolution, near-real-time data ingestion pipeline. The existing prototype's Data Sources section [Image 2] already lists sources such as ACE, which is a core contributor to the OMNI2 dataset cited in the research paper. This existing infrastructure provides a direct path for the new data pipeline.

The system will be configured to ingest data from space missions that contribute to the OMNI2 dataset, with a specific focus on those providing high-resolution measurements of the key predictor variables identified in the academic paper. The existing ACE data source [Image 2] will be prioritized and integrated with a data parser for its Solar Wind (SW) Plasma and Interplanetary Magnetic Field (IMF) data streams. This will directly supply the Scalar B, Proton Density, Plasma Speed, and Proton Temperature variables, which were identified as crucial for the Random Forest model's performance. A new API endpoint, provisionally named /api/data/ingest, will be established to handle the continuous stream of data.

A fundamental distinction must be made between the methodology of the academic paper and the requirements of an operational forecasting system. The paper's conclusions on the superiority of the Random Forest model were based on a study of data resampled into 30-day intervals. This low-resolution approach is effective for analyzing long-term trends and historical event frequency, but it is insufficient for the user's requirement for 3-hour and 30-minute forecasts. A robust operational system must be capable of predicting transient, short-term phenomena. This requires a dedicated data engineering effort to process and prepare raw, high-resolution data at a granular level, such as hourly or sub-hourly intervals. By training a new Random Forest model on this high-frequency data, the system can learn the transient dynamics required for accurate short-term forecasting. The model's efficacy is therefore not a simple replication of the paper’s findings, but a strategic application of its validated methodology and identified variables to a new, higher-resolution dataset. This approach ensures the final product is both scientifically grounded and operationally relevant.

The raw data will be resampled and processed to create a training dataset suitable for the forecasting model. For the 3-hour forecast, a rolling window of the last 3 to 6 hours of data will be used to predict the Dst value in the next 3 hours. For the 30-minute forecast, a more granular window of 30 to 60 minutes will be used. The Dst-index, which serves as the target variable for prediction, will also be used as a key feature (Dst-index MIN) from the look-back window. The model will use the Dst values from the look-back window, along with the other precursor variables, to predict the future Dst and storm intensity. I.2 Machine Learning Model Implementation
Based on the academic paper's findings, the Random Forest Regressor is the most effective model for this application.

The core model will be implemented using the scikit-learn library. Hyperparameter optimization, as described in the research paper's methodology, will be conducted using GridSearchCV or a similar automated approach to optimize the model's performance on the new high-resolution training data. The model will be explicitly trained on the variables identified in the paper as most important for prediction, including Dst-index MIN, Scalar B, Alpha/Proton ratio, R (Sunspot No.), and Solar Wind Plasma Temperature. This selective approach ensures the model is lean and focuses on the most predictive inputs.

It is noted in the academic paper that the Dst-index MIN is the most important feature, accounting for over 30% of the model's predictive power. A superficial interpretation of this finding might lead to the erroneous conclusion that the model is circular. However, in the context of a real-time forecasting system, this finding is crucial. The model’s true predictive power for a forward-looking forecast comes from its ability to use the current state of physical precursor variables, such as Scalar B (26.5% importance) and Alpha/Proton ratio (16.2% importance), to predict the future Dst value. The paper's analysis of feature importance validates the physical causal link between these precursor variables and the resulting storm intensity, a critical piece of information for constructing a reliable predictive engine. This shifts the model's function from a statistical analysis of historical events to a real-time predictive engine that uses the physical drivers of a storm to anticipate its effects.

I.3 API Endpoint Specification
New API endpoints will be created to expose the trained model's functionality to the frontend, ensuring seamless data flow and operational control.

GET /api/forecast/geomagnetic/{horizon}: This endpoint will return a JSON object with the latest prediction for the specified forecast horizon (e.g., 30min, 3hour). The response will include the predicted Dst value, a corresponding confidence score, and a timestamp.

POST /api/model/train/geomagnetic: This endpoint will trigger a manual retraining of the geomagnetic storm model. It will accept parameters for the training period and optional hyperparameters, providing operators with direct control over the model's lifecycle.

GET /api/model/metrics/geomagnetic: This endpoint will return a JSON object containing the latest performance metrics of the model (e.g., R² score, RMSE, Feature Importance) from the most recent training run. This data will be used to dynamically populate the Model page, providing transparency and performance insights.
